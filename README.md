## 🧭 기획 배경

가계부 앱은 많지만, 단순히 "기록하고 끝"인 경우가 대부분이었습니다.

- 내 수입과 지출 흐름을 보면서 **"그래서 나 지금 투자 해도 되나?"** 라는 질문이 자연스럽게 떠올랐습니다.
- 재무 상담은 접근 장벽이 높고, 투자 앱은 내 가계부 데이터를 모릅니다.
- 그래서 **가계부와 AI 투자 조언을 한 화면에서** 해결하는 앱을 직접 만들기로 했습니다.

"기록 → 파악 → 결정"의 흐름을 한 곳에서 완성하는 것이 목표였습니다.

---

## ✨ 주요 기능

**가계부**
- 수입 / 지출 거래 내역 등록, 수정, 삭제
- 총수입 · 총지출 · 현재 잔액 자동 계산
- 최근 월별 수입 / 지출 / 잔액 요약 표시
- 거래 데이터는 브라우저 `localStorage`에 저장 (서버 불필요)

**AI 투자 조언**
- 투자 성향 · 목표 · 기간을 입력하면 Gemini AI가 맞춤 조언 생성
- 현재 가계부 요약(총수입, 총지출, 최근 6개월 흐름)을 자동으로 AI에게 전달
- 조언에는 분산 투자 · 리스크 · 비상금 등 현실적인 관점 포함

## 🛠 기술 스택

### 선택의 이유까지

| 영역 | 기술 | 선택 이유 |
|------|------|-----------|
| Frontend | React 19 + Vite 8 | 최신 React Compiler 실험 겸 빠른 개발 환경 |
| 아이콘 | lucide-react | 가볍고 디자인 일관성이 좋음 |
| Backend | Node.js (기본 `http` 모듈) | 의존성 없이 서버 구조를 직접 이해하고 싶었음. Express 없이 처리 가능한 규모 |
| AI | Google Gemini API (`gemini-2.5-flash`) | 무료 할당량 안에서 빠르게 응답하는 모델 |
| 배포 | Netlify (프론트) + Render (백엔드) | 무료 티어로 풀스택 배포 가능 |
| 데이터 저장 | 브라우저 `localStorage` | 개인 용도이고 서버 비용 없이 빠르게 구현하기 위한 의도적 선택 |

> **Express를 쓰지 않은 이유:** 라우팅이 `/api/health`, `/api/investment-advice` 두 개뿐이라 프레임워크 없이 직접 구현했습니다. HTTP 서버의 동작 원리를 코드 레벨에서 이해하는 것도 목적 중 하나였습니다.

## 🔥 트러블슈팅

### Netlify 배포 후 API 요청이 전부 실패

**상황**
로컬에서는 잘 동작하던 AI 조언 요청이, Netlify 배포 직후 전혀 작동하지 않았습니다.

**원인 분석**
로컬에서는 Vite의 프록시 설정 덕분에 `/api/investment-advice`라는 상대 경로가 자동으로 `localhost:3001`로 포워딩됩니다.  
하지만 Netlify에 올라간 정적 파일은 프록시가 없어서, `/api/...` 요청이 Netlify 자체로 향했고 404가 반환됐습니다.

```
// 문제가 된 코드
fetch('/api/investment-advice', ...)
// → 로컬: Vite가 localhost:3001로 프록시 ✅
// → Netlify: Netlify 서버로 직접 요청 ❌
```

**해결**
환경을 자동으로 감지해서 API 베이스 URL을 분기하는 함수를 추가했습니다.

```js
function resolveApiBaseUrl() {
  const configured = import.meta.env.VITE_API_BASE_URL?.trim();
  if (configured) return configured.replace(/\/+$/, '');
  if (import.meta.env.DEV) return ''; // 개발: 상대 경로 → Vite 프록시
  return 'https://two026-04-13money.onrender.com'; // 프로덕션: Render URL
}
```

**배운 것**
로컬 프록시는 개발 편의를 위한 장치일 뿐, 프로덕션과 동일하지 않습니다.  
배포 환경을 처음부터 고려한 API URL 설계가 필요하다는 것을 체감했습니다.

### Render 서브도메인 이름 문제

**상황**
백엔드 서비스 이름을 `2026-04-13money`로 짓고 싶었지만, Render가 숫자로 시작하는 서브도메인을 허용하지 않았습니다.

**해결**
`two026-04-13money`로 변경해 배포했습니다 (`2` → `two`).

**배운 것**
도메인/서브도메인 네이밍 규칙(영문자 시작)은 플랫폼마다 다를 수 있습니다.  
배포 전에 네이밍 규칙을 확인하는 습관이 필요합니다.

## 📝 회고 / 배운 점

**잘한 것**

- Express 없이 Node.js 기본 모듈로 서버를 구현해보면서, HTTP 요청·응답의 흐름을 더 구체적으로 이해하게 됐습니다.
- 데이터 저장을 `localStorage`로 결정한 것도 "개인 용도에서 가장 빠른 방법"을 선택한 의도적인 트레이드오프였습니다.
- 프론트-백을 분리 배포하면서 CORS, 환경별 URL 분기 등 실제 운영 이슈를 직접 부딪혀 봤습니다.

**아쉬운 것 / 다음엔 이렇게**

- 거래 데이터를 `localStorage`에만 저장하다 보니, 브라우저 데이터 삭제 시 기록이 사라집니다. 실용적으로 쓰려면 서버 저장소나 클라우드 동기화가 필요합니다.
- 루트 `.gitignore`가 없어서 `backend/.env` 파일이 실수로 커밋될 위험이 있었습니다. 다음 프로젝트에서는 초기 설정 시 반드시 챙기겠습니다.
- 커밋 2개로 끝난 것처럼 보이지만, 실제로는 기능 단위로 커밋을 분리하는 연습이 필요하다는 것을 느꼈습니다.
